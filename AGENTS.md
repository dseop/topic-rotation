# Repository Guidelines

## Project Structure & Module Organization
- Source: root TypeScript modules (`index.ts`, `writingTopics.ts`).
- Tests: Jest-style specs in `__tests__/` (e.g., `writingTopics.test.ts`).
- Build output: `dist/` (generated by TypeScript).
- Public API: re-exported from `index.ts`. Add new modules under root and export them via `index.ts`.

## Build, Test, and Development Commands
- `npm run build`: Compile TypeScript using `tsconfig.json` to `dist/`.
- `npm test`: Currently a no-op. To enable tests, install Jest and a TS preset, then run.
- Example local run after build: `node -e "console.log(require('./dist').getTopicsCount())"`.
- Pack for distribution: `npm pack` produces a tarball (e.g., `*.tgz`).

## Coding Style & Naming Conventions
- TypeScript, 2-space indentation, semicolons, double quotes.
- Types/Interfaces: `PascalCase` (e.g., `WritingTopic`).
- Functions/variables: `camelCase` (e.g., `getRandomWritingTopic`).
- Constants: `UPPER_SNAKE_CASE` (e.g., `WRITING_TOPICS`).
- Keep modules small and focused; export only what’s public from `index.ts`.

## Testing Guidelines
- Framework: Jest (tests live in `__tests__/`, `*.test.ts`).
- Suggested setup: `npm i -D jest ts-jest @types/jest` and init with `npx ts-jest config:init`.
- Run: `npm test` after wiring the Jest script (e.g., `"test": "jest"`).
- Add deterministic tests for date-based helpers (`getTodaysTopic`, `getTopicByDate`).

## Commit & Pull Request Guidelines
- Use Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, etc. Example: `feat: add getTopicsCountByCategory`.
- Keep commits small and focused; include context in body when needed.
- PRs must include: summary, motivation/linked issue, before/after notes or examples (e.g., sample usage), and any testing steps.
- Ensure CI/build passes locally (`npm run build`) and tests (once enabled).

## Security & Configuration Tips
- Do not commit build artifacts or locks; `.gitignore` excludes `dist/`, lockfiles, and secrets.
- No secrets required; avoid adding `.env` files. If needed, document variables in README and keep examples in `.env.example`.
